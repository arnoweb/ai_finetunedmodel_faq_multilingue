Features

What the application actually does.

Natural language search

Users ask their question normally — synonyms, rephrasing, and typos don't stop the right answer from being found.

Bilingual FR / EN

One engine, language picked at runtime — not two separate FAQs to maintain.

Confidence score & refusal when unsure

Every result shows its similarity score; below the confidence threshold, the app says so clearly instead of guessing.

Optional RAG generation

A written answer, built strictly from the retrieved documents, one click away.

Self-hostable

Runs on your own infrastructure, with no mandatory dependency on a closed platform.

Fine-tuned on your data

The model learns your FAQ's own vocabulary, not a generic one.

Business value

A static FAQ fails the moment the user doesn't type the exact words the support team chose — the example above proves it: "forgot my password" and "reset" share no common word, yet it's the right answer.

Keyword-based FAQ
  • "I forgot my password, what do I do?" returns nothing if the FAQ says "reset my password"
  • No tolerance for typos, synonyms, or rephrasing
  • No quality signal — no way to know if the results are actually relevant
  • Same experience regardless of the user's language
Fine-tuned semantic retrieval
  • Compares the meaning of the question, not the exact words
  • Returns a similarity score — and declines to answer below a set threshold
  • Measured objectively (MRR, Recall@K) on a dedicated evaluation set
  • One engine for both French and English, no duplicated logic

06

Why a fine-tuned model rather than an AI agent?

Many FAQ solutions rely on an AI agent that reasons through every single question. This project makes a different choice, deliberately: a specialized search engine to find the answer, an optional LLM only to write it up.

AI agent for every question
  • An LLM call on every search — cost and multi-second latency
  • Non-reproducible: the same question can return two different answers
  • Quality is hard to measure objectively
  • FAQ content is systematically sent to a third party, on every question
Specialized model (this project)
  • Search in 10–50 ms, locally, no network call
  • Reproducible and measurable results (MRR, Recall@K)
  • The LLM only steps in at the end, on request, to write the answer
  • No data sent to a third party for the search step itself

Does the model need to be fine-tuned every time the FAQ changes?
No. The model turns any content into search vectors, including a question added a minute ago — a fast operation, recomputed continuously (every 5 minutes at most). Fine-tuning addresses a different need: sharpening the model's precision on the client's domain vocabulary, over time.

MechanismFrequencyRole
FAQ vector refreshContinuous (≤ 5 min)Makes new content immediately findable
Model fine-tuningPeriodic — new vocabulary, growing volume, or slipping metricsImproves domain vocabulary understanding
Freshness guarantee: any FAQ content addition or edit is findable in search within 5 minutes, with no restart and no fine-tuning — provided the source content (CMS or otherwise) is synced to the FAQ files the application reads.
